## 1. 节点预售 vs V 级体系（最重要的不一致）

### 现状

链上 `NodePresell.sol` 的 `directRate` 是**节点档位级的**：

| nodeId | 档位 | 单价 USDT | directRate |
|---|---|---|---|
| 101 | 联创·符主 | 50,000 | 15% |
| 201 | 超级·符魂 | 10,000 | 12% |
| 301 | 高级·符印 | 5,000 | 10% |
| 401 | 中级·符源 | 2,500 | 8% |
| 501 | 初级·符胚 | 1,000 | 5% |

下线购节点时，链上把 `payAmount × directRate / 10000` 直接转给上线钱包（不经合约托管）。

### 规范文档要求

§5：
- 直推奖励**固定 5%**
- 团队奖励按 V1-V9 级差 **4%-29%**

### 差距

1. **直推**链上是浮动（5–15%），spec 是固定 5%。要么改合约（重新部署 + 迁移现有数据），要么改 spec（认可现实）。
2. **团队**链上无任何"V 级"概念，奖励只在购买瞬间转给上线，不存在"层层级差结算"。

### 处理策略

- M0 已把两套口径都写入 `system_config`（`general.directRate.specPct=5`，`node_tier.tier_*.directRatePct=5..15`）。
- M3 在 admin 的 Contracts / Rewards 页同时显示**两套数字**，红色高亮差异。
- DB function `compute_team_commission` 按 V 级算"应给"，对照已上链的"实给"，差额作为运营决策依据。

---

## 2. 套餐 / 三池 / 日化 完全缺失

spec §2-§4 描述的是一个**理财类协议**：用户存 USDT → 拆三池 → 每日产生收益。

链上**完全没有这套合约**。现有合约只做了节点预售，离 spec 描述的产品形态有相当距离。

### 处理策略（产品决策待你定，非工程问题）

- 若 V2.0 计划上线套餐合约：应起一份独立 spec（套餐合约 ABI、三池合约地址、结算合约 cron 触发方式），后端起 `package_orders` / `daily_settlements` 等表，admin 加"套餐管理"页。
- 若不上线：把 spec §2-§4 标为「设计参考，未启动」，admin 仅作参数展示。
- M0 已把所有 spec 参数写入 `system_config` 备查。

## 5. 给运营 / 产品的关键问题清单

1. **直推率**到底按规范的固定 5% 还是按合约的浮动 5–15%？决定后要么改合约，要么修 spec。
2. **V 级团队奖励**何时上线？是新合约 + 全量迁移，还是 off-chain 计算 + 周期性发放？
3. **套餐 / 三池 / 日化**是 V2.0 路线图，还是 spec 仅为愿景？影响 admin 是否要预留页面入口。
4. **DAO 治理**（V9 解锁）需要单独治理合约（如 OpenZeppelin Governor）吗？
5. **节点空投 1000 万 RUNE**是一次性活动还是持续奖励？合约层是否要起独立 airdrop 合约？

回答上述会决定 admin 后台是否要再加 4-5 张页面（套餐 / 空投 / 治理 / V 级结算队列）。
