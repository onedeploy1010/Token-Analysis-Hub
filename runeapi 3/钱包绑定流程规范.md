# 钱包连接 · 推荐人绑定流程规范

> 适用范围：`crypto-analyzer` 前端（`rune-ai.io` / `rune-testnet.pages.dev`）
> 涉及合约：`Community.addReferrer(address)` / `Community.referrerOf(address)`
> 涉及表：`rune_referrers`、`rune_members`（由 `api-server/indexer/rune-indexer.ts` 写入）
> 版本：v1 · 2026-04-30

---

## 1. 总则

1. **「连接钱包成功」的唯一判定标准是：链上 `referrerOf(user) != 0x0`**。仅完成钱包签名/账户授权但未绑定推荐人的状态，不算连接成功，UI 不得进入主应用（dashboard / profile 等）。
2. 绑定推荐人是一笔上链交易（`Community.addReferrer`），交易确认后由后端 indexer 自动写入数据库（`rune_referrers` + `rune_members`）。前端**不得自行写库**，所有 DB 记录必须以链上事件为唯一来源。
3. 合约层面已强制：上级必须为 ROOT (`0x…0001`) 或自身已绑定的地址，且新用户尚未绑定。因此**只要 `addReferrer` 交易成功上链，绑定就一定存在**——前端任何时候都不应把"查不到"当作"绑定失败"展示给用户。

---

## 2. 连接钱包的状态机

```
disconnected ──connect()──▶ wallet-signed
                                 │
                                 ▼
                       readReferrerOf(user)
                                 │
              ┌──────────────────┼──────────────────┐
              ▼                  ▼                  ▼
        != 0x0  →  bound      == 0x0  →  unbound    error → retry/disconnect
              │                  │
              ▼                  ▼
        进入主应用          弹 BindReferrerModal
                                 │
                                 ▼
                    用户提交 addReferrer 交易
                                 │
                                 ▼
                          confirming（见 §4）
                                 │
                                 ▼
                         bound → 进入主应用
```

**禁止**：在 `wallet-signed` 但 `unbound` 的状态下放行用户进入 dashboard / profile / 任何带返佣或团队数据的页面。允许停留的只有：BindReferrerModal、`/recruit`（销售页）、纯静态文档页。

**关闭绑定弹窗的处理**：用户点击 X / Esc / 弹窗外侧关闭 BindReferrerModal 时，应**断开钱包**（`disconnect(wallet)`），UI 退回到未连接状态。原因：连接但未绑定的"半连接"状态没有任何可用功能，会让用户困惑。

---

## 3. 绑定 → 数据库的同步链路

绑定不是前端写库，而是**链上事件 → indexer → DB** 的单向流：

| 阶段 | 触发 | 写入位置 | 延迟 |
|---|---|---|---|
| ① 链上 | `Community.addReferrer(referrer)` 上链 | 链上 storage + emit `EventAddReferrer(user, referrer)` | 用户钱包签名后即时（1 个 BSC 块 ≈ 3s） |
| ② Indexer | `api-server/indexer/rune-indexer.ts` 轮询 `getLogs` | `rune_referrers`（事件原始记录）+ `rune_members`（成员索引） | **轮询间隔 15s + 3 块确认缓冲 ≈ 9s**，理论上限 ≤ 25s，实际通常 < 30s |
| ③ GraphQL / REST | `api-server` 读 `rune_referrers` / `rune_members` | — | 读取即时 |

**前端不得**：
- 在 `addReferrer` 交易确认回调里调用任何 `POST /api/...` 写库接口
- 直接通过 Supabase SDK 往 `rune_referrers` 或 `rune_members` 插数据
- 自行维护"已绑定"状态的本地缓存超过单次会话寿命

唯一允许的「绑定完成」判定：**重新读链上 `referrerOf(user)` 返回非零地址**。DB 记录的可见性比链上读慢一个 indexer 周期，所以「绑定后立刻进 dashboard」必须以链上读为准，不能等 DB。

---

## 4. 轮询窗口的 UI 文案规范

绑定交易确认后，前端会在一个时间窗口内反复重读 `referrerOf` / 查询 GraphQL，直到拿到非零结果。**这个窗口里 UI 必须显示"正在查找"语义，禁止任何"失败 / 未绑定 / 错误"措辞。**

**理由**：合约已经保证绑定成功的前提条件。交易已上链确认即意味着绑定一定存在，"查不到"只是 RPC 节点的同步延迟或 indexer 还没追上区块——不是失败。把延迟当失败展示给用户，会让用户误以为合约出错而重复发起 `addReferrer`（第二次会被合约 revert，浪费 gas + 增加客服负担）。

### 4.1 文案对照

| 场景 | ✅ 应显示 | ❌ 禁止显示 |
|---|---|---|
| `addReferrer` 交易已发出，等待上链 | `提交中…` / `Submitting…` | `失败` / `Failed` |
| 交易确认后 ≤ 30s，`referrerOf` 仍为 `0x0` | **`正在链上查找…`** / **`Verifying on-chain…`** | `未绑定` / `Not bound` / `Failed` |
| 交易确认后 30~120s，仍为 `0x0` | `正在链上查找…（节点同步中）` / `Verifying on-chain… (node syncing)` | `失败` / `重试` |
| > 120s 仍 `0x0` 且 `getTransactionReceipt` 状态非 success | `交易未上链，请重试` + 重新发起按钮 | — |
| 用户未连接钱包 | `请连接钱包` | `绑定失败` |

### 4.2 轮询参数（建议）

```ts
{
  initialDelayMs: 1500,      // 给 RPC 节点 1.5s 缓冲
  intervalMs:     2000,       // 每 2s 重读一次 referrerOf
  maxAttempts:    60,         // 共 ~120s 窗口，覆盖 indexer 一个完整轮询周期
  onAttempt:      () => "正在链上查找…",
  onTimeout:      () => "交易未上链，请重试",
}
```

### 4.3 失败的真正定义

**只有以下两种情况才允许向用户展示「失败」**：

1. `sendTransaction` 抛错（钱包拒签、gas 不足、合约 revert）。此时 `addReferrer` 根本没上链，让用户重试是合理的。
2. 轮询窗口超时后，`getTransactionReceipt(txHash)` 返回 `status !== "success"`。此时是真实的链上失败（reorg / revert）。

其他一切场景（RPC 暂时返回旧值、indexer 还没追上、GraphQL 缓存未刷新）都属于**同步延迟**，UI 必须保持"正在链上查找"的中性等待状态。

## 6. 边界与例外

1. **ROOT 地址**（`0x0000…0001`）：合约层面允许其作为绑定上级，但 ROOT 自身不会出现在 `rune_members`（因为它从未触发 `EventAddReferrer`）。前端不应把 ROOT 当作"普通用户"展示在团队/推荐链中。
2. **演示模式（demo mode）**：`/demo?demo=<nodeId>` 路径绕过本规范全部约束，直接放行进入 dashboard。该模式仅用于教学/截图，前端必须通过 `useDemoStore.isDemoMode` 判定，且 demo 状态不得与真实钱包状态混用。
3. **链切换**：用户从 BSC 主网切到测试链（或反之）时，必须重新跑一遍 §2 的状态机，因为 `referrerOf` 是按 chainId 隔离的——主网绑定不等同于测试网绑定。
4. **多钱包**：同一浏览器切换钱包账户时，`address` 变化即视为新会话，所有 dismiss 标志、轮询计时器必须重置。

## 7. 验收清单

发布前手工跑一遍：

- [ ] 全新钱包连接 → BindReferrerModal 自动弹出
- [ ] 关闭 BindReferrerModal → 钱包自动断开
- [ ] 用一个未注册的地址作为推荐人 → 弹窗显示"上级地址尚未在社区注册"（不是"失败"）
- [ ] 用 ROOT 地址 (`0x0000…0001`) 作为推荐人 → 通过校验
- [ ] 提交绑定交易，签名后 5s 内 → UI 显示「正在链上查找…」（非「失败」）
- [ ] 交易上链 ≤ 30s → 自动跳转 dashboard
- [ ] 进入 dashboard 后 ≤ 30s → 团队 / 返佣 GraphQL 数据出现（indexer 已追上）
- [ ] 已绑定地址重新连接 → 跳过 BindReferrerModal，直接进 dashboard
- [ ] 切换钱包账户到未绑定地址 → 重新弹 BindReferrerModal
